## 1. Бизнес-цель и ожидаемый результат (ТЗ)

**Проблема:** ранние сигналы стресса ликвидности на российском денежном рынке разбросаны по множеству публичных источников; ручной мониторинг медленный и не масштабируется.

**Цель:** автоматизировать сбор данных, рассчитать сигналы по **пяти независимым модулям**, агрегировать их в единый **Liquidity Stress Index (LSI, 0–100)** с интерпретацией вклада каждого модуля, статусами (зелёный / жёлтый / красный) и **дашбордом**. Технологически ТЗ задаёт экосистему **Python**: pandas, scipy, scikit-learn; пересчёт данных **не реже одного раза в сутки**.

**Минимум по ТЗ:**

- все пять модулей выдают нормализованные сигналы;
- агрегация — ML по выбору команды с **обязательной интерпретируемостью** (SHAP, feature importance или явная формула);
- дашборд: LSI, алерты, графики;
- **backtest** на эпизодах: декабрь 2014, февраль 2022, август 2023 (ориентир — показатели из таблицы «Ликвидность банковского сектора» ЦБ как ground truth).

**Максимум (бонус +15 баллов):** LLM — автоматический текстовый комментарий при пересчёте и интерактивный чат на отдельной вкладке дашборда; базовый уровень бонуса — **RAG только по данным системы**.

**Пайплайн по ТЗ (логическая схема):**

`M1…M5 → MAD-нормализация → агрегационный слой (ML) → LSI + статус + объяснение → (бонус) LLM`.

**Нюансы ТЗ:**

- **M4** в ТЗ задуман как **мультипликатор Seasonal_Factor (1.0–1.4)** на агрегационном слое; отмечена **исследовательская задача** по пересечению сигналов с M1/M2/M5.
- **Валидация:** чувствительность весов ±20%, отложенная выборка для backtest.

## 3. Что реализовано в `final.ipynb` (контракт данных и логика)

### 3.1. Объединение модулей

- **Ключ:** `date` (нормализованный календарный день).
- **Календарь:** `pd.date_range` от минимальной до максимальной даты среди всех модулей; к базе последовательно делается `left merge` таблиц M1–M5.
- **Префиксы колонок:** `m1_`, `m2_`, `m3_`, `m4_`, `m5_` для различения источников.
- **Пропуски:** на днях без аукционов репо признаки M2 остаются NaN (в выводах ноутбука отмечена высокая доля NaN по M2 — ожидаемо для дневного ряда).

### 3.2. M1 — усреднение обязательных резервов + RUONIA

- **Источники:** Excel ЦБ по обязательным резервам; выгрузка RUONIA через URL `UniDbQuery` ЦБ.
- **Логика:** спред `m1_shift = m1_fact - m1_need`; MAD-нормализация спреда и RUONIA на **окне ~3 года**; флаг конца периода усреднения **`m1_Flag_EndOfPeriod`** (последние дни до 14-го числа месяца по правилу из кода).
- **Частота:** дневной ряд после объединения с RUONIA.

### 3.3. M2 — репо ЦБ (фокус на 7-дневных аукционах)

- **Режимы:** долгий обход страниц ЦБ по датам (`M2_REPO7D_REFRESH_FROM_WEB = True`) с кешированием в `m2_df_m2_cache.csv` **или** загрузка из CSV (`False`).
- **Показатели:** спрос/размещение, ставки, **Cover ratio**, спред ставки к ключевой, **`m2_Flag_Demand`** (переспрос), **MAD_score_cover**, **MAD_score_rate_spread** (окно MAD в коде — порядка 3 лет, см. функцию `calc_mad_score`).
- **Особенность для архитектуры:** отдельный **ETL-планировщик** (например, только дни проведения аукционов или известный график «вторники») + хранение сырья и агрегатов, чтобы не опрашивать весь диапазон дат ежедневно.

### 3.4. M3 — размещение ОФЗ (Минфин)

- **Источники:** набор фиксированных URL по годам на xlsx итогов аукционов Минфина.
- **Обработка:** нормализация схем колонок по годам, расчёт **cover_ratio**, флаги **недоспроса/переспроса**, **yield_spread** относительно «кривой» по дням до погашения внутри дня аукциона; агрегация на уровень даты; **дневной ряд с ffill** между датами аукционов.
- **MAD:** для `cover_ratio` (с clip), `yield_avg`, `yield_spread` на окне **756 дней** для части истории (с 2021 — в коде отдельная ветка нормализации).

### 3.5. M4 — налоговый календарь (ФНС)

- **Источник в коде:** фиксированный URL open data ФНС (`data.nalog.ru/...kalendar...xml`).
- **Логика:** разбор XML, извлечение «важных» налогов по правилам regexp к заголовкам, dummy по типам налогов, счётчик **`m4_n_important`**.
- **На агрегации:** производные **`Tax_Week_Flag`** (скользящее окно 7 дней по числу важных событий) и тригонометрическая кодировка дня года **`Seasonal_Factor_sin/cos`** (не тот же самый **Seasonal_Factor 1.0–1.4** из ТЗ — см. раздел «разрывы»).

**Риск для архитектуры:** календарь в примере покрывает **ограниченный набор лет** из пакета XML; для полного исторического backtest нужна стратегия **нескольких выпусков XML по годам** или другой официальный API/файлы.

### 3.6. M5 — «казначейство» / ликвидность по данным ЦБ

**Фактическая реализация** опирается на HTML-таблицу **`bliquidity`** (раздел ЦБ HD Base), парсинг через `pandas.read_html`, признаки:

- дефицит ликвидности, корсчета, обязательные резервы, производные **`treasury_pressure`**, MAD-оценки **`m5_MAD_score_treasury_pressure`**, **`m5_MAD_score_liquidity_deficit`**, флаг **`m5_flag_stress`** (правило по порогам MAD и концу месяца).

**Отличие от ТЗ:** в задании перечислены также **Roskazna (размещения ЕКС)** и **СОРС ЦБ («привлечённые средства»)**; в текущем контуре они **не подключены** как отдельные ряды — при проектировании архитектуры нужно решить: расширить M5 до полного ТЗ или зафиксировать упрощение с обоснованием.

## 4. Агрегационный слой LSI (как сделано сейчас)

### 4.1. Teacher (опорная метка)

Так как **нет отдельной размеченной цели LSI**, строится **эвристическая цель `LSI_teacher`**: выпуклая комбинация **MAD-сигналов** (через `relu` и cap) и **весовых флагов** (M1 end-of-period, M2 demand, M3 недоспрос, M5 stress, Tax week). Затем масштабирование на диапазон **[0, 100]**.

Это соответствует духу ТЗ (интерпретируемая агрегация), но **не заменяет** требование backtest против **официальных рядов ликвидности** ЦБ.

### 4.2. Модели

- Ансамбль: **HistGradientBoostingRegressor** (несколько конфигураций), **GradientBoostingRegressor**, опционально **CatBoostRegressor**; усреднение предсказаний → **`LSI_ensemble`**, клип **[0, 100]**.
- Опционально: **DLinear** через обёртку к `Time-Series-Library` (последовательность признаков + teacher как часть окна).

### 4.3. Интерпретируемость

- Усреднение **`feature_importances_`** по моделям дерева решений и вывод топ-признаков (не SHAP; ТЗ допускает feature importance как достаточный вариант).
- Явный список колонок признакового вектора `X` документируется в ноутбуке.

### 4.4. Статусы стресса (как на графиках)

Условные зоны по ТЗ используются в визуализации ноутбука:

- зелёный: **0–40**
- жёлтый: **40–70**
- красный: **70–100**

(Пороги стоит вынести в конфиг дашборда/алертинга.)

### 4.5. Дополнительный анализ в ноутбуке

- Сравнение `LSI_teacher` и `LSI_ensemble`, сглаживание (Savitzky–Golay), **seasonal_decompose** (additive, период 7), регрессия остатков на признакки качества данных — полезно как прототип **мониторинга качества модели**, но не как production-сервис.

## 5. Разрывы между ТЗ и текущей реализацией (для backlog архитектуры)

| Область | ТЗ | Сейчас |
|--------|-----|--------|
| **Доставка данных** | Плановый сбор ≥1 раз/сутки | Ручной/интерактивный запуск ноутбука |
| **M4 Seasonal_Factor** | Множитель **1.0–1.4** на слое агрегации | Sin/cos + Tax week flag в признаках; нет явного множителя из ТЗ |
| **M4 горизонт** | Полный календарь истории для merge | Зависит от выбранного XML; может не покрывать все годы широкой таблицы |
| **M5** | ЦБ СОРС + Roskazna + калибровка | В основном **bliquidity** ЦБ |
| **Backtest** | Обязательные эпизоды 2014, 2022, 2023 vs ground truth ЦБ | В ноутбуке есть визуальные аннотации периодов, **формального backtest-отчёта по метрикам ТЗ нет** |
| **Интерпретация** | SHAP допустимо как усиление | Преимущественно **feature importance** |
| **Дашборд и алерты** | Веб-дашборд, алерты | Только matplotlib / табличный вывод в ноутбуке |
| **LLM / RAG** | Бонус | Не реализовано |

## 6. Рекомендуемые компоненты целевой архитектуры (черновик для проектирования)

Ниже — логическое разбиение без привязки к конкретному облаку; его можно наложить на микросервисы или монолит.

### 6.1. Слой сбора данных (Ingest)

- Отдельные **коннекторы** на источник: ЦБ (xlsx/html/query), Минфин (xlsx), ФНС (XML), при расширении M5 — Roskazna, СОРС.
- **Идемпотентность**, **ретраи**, **версионирование сырых файлов** (объектное хранилище + метаданные даты загрузки).
- Расписание (**cron**, Airflow, Prefect и т.д.) с частотами по модулю (ТЗ: M1 месяц, M2 день, M3 несколько раз в неделю, …).

### 6.2. Слой нормализации и feature store

- Единый **календарь** и справочник «день торгов vs выходной» при необходимости.
- Детерминированные трансформации MAD, флаги — **чистые функции** (unit-тесты на фикстурах).
- Сохранение **широкой дневной таблицы** (аналог `wide` / `wide_lsi`) в **Parquet** или БД временных рядов.

### 6.3. Слой модели (Training vs Serving)

- **Обучение:** офлайн-джоб с отчётами (metrics, calibration plots, sensitivity ±20%).
- **Сервинг:** версия модели в registry; API **POST/GET** «LSI на дату» + «вклад признаков/модулей».
- Разделить **teacher** как эталон для обучения и **продактовый scoring**, если решите уйти от teacher к чистому supervised по ряду ЦБ.

### 6.4. Объяснение и мониторинг

- Для каждого скора: вклад по **группам признаков M1–M5** (агрегация SHAP/importance по префиксам).
- Дрейф признаков, доля NaN по модулям, задержки ETL — **дашборд качества данных**.

### 6.5. Приложение «Дашборд»

- Вкладки: **текущий LSI**, история, разрез по модулям, сырые ключевые графики (спред+M1, cover repo, cover ОФЗ, налоговые окна, M5).
- **Алерты:** правила по порогам LSI и по комбинациям флагов; каналы (email, мессенджер) — на выбор.

### 6.6. LLM (бонус)

- **Автокомментарий:** structured prompt с числовыми полями последнего пересчёта.
- **Чат:** RAG по векторному индексу документов = снапшоты wide-таблицы, описания модулей, налоговый календарь; запрет на «галлюцинации» вне корпуса на базовом уровне ТЗ.

### 6.7. Безопасность и соответствие

- Публичные данные; при развёртывании в банке — учёт **политики исходящего доступа**, хранения логов, запрета утечки промптов/PII.

## 7. Зависимости и внешние риски

- **Хрупкость парсинга:** HTML/Excel ЦБ и Минфина меняют вёрстку — нужны **тесты на схему** и алерты при поломке.
- **Производительность M2:** полный веб-скрейп по датам неприемлем в prod — нужен **инкрементальный** сбор и кеш.
- **Юридическое использование данных:** опираться только на **официальные** open data и условия использования.
- **Time-Series-Library:** лицензия и обновления upstream; для prod можно зафиксировать commit или заменить DLinear собственной лёгкой реализацией, если нужна минимальная поверхность.

6. (Бонус) **LLM + RAG** по данным системы и отдельная вкладка чата.
